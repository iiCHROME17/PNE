# Chapter 4: Methodology

## 4.1 Data Collection

The primary data collection strategy was qualitative and iterative, centred on the systematic examination of structured output logs produced by the system during test conversations. Each full conversation session was automatically serialised to a timestamped JSON file — such as `amourie_oth0204.json`, a log of a four-turn exchange with the guard character Amourie Othella in the `door_guard_night` scenario — providing a complete record of every layer of the BDI pipeline's execution. These logs captured the NPC's internal thought, subjective belief, emotional valence, desire state, behavioural intention, interaction outcome, and final NPC attribute values, alongside the player's choice text, language art classification, and dice check results.

The purpose of this logging strategy was diagnostic transparency rather than quantitative measurement. The rich JSON structure allowed each turn's pipeline state to be read alongside the NPC's spoken response, making dissonances — cases where the pipeline's internal reasoning was inconsistent with the generated dialogue, or where NPC attributes drifted implausibly — directly legible without requiring code instrumentation. The collection process was primarily trial-and-error: a test scenario was designed, the engine was run with a defined set of player choices, and the resulting JSON was examined — first by the developer and subsequently with the assistance of an AI language model (Claude, Anthropic) — to identify narrative dissonance or technical faults. Dissonance was defined qualitatively as any case where the NPC's response was incongruent with its internal state, where an outcome label did not correspond to response content, or where final NPC state values appeared implausible. Technical faults included malformed JSON keys, missing terminal outcome fields, and desire states that matched no entry in the intention registry.

The logs did not support formal statistical analysis — sample sizes were too small and conditions too variable — but were highly effective for identifying structural and logical failures. Quantitative data was incidentally present in the form of numeric attribute values (e.g. `emotional_valence: -0.14`, `confrontation_level: 0.593`), used informally to confirm that deltas were accumulating in the expected direction. One specific quantitative observation did directly inform a design revision: response latency. During early development, the cognitive layer used a live call to a locally hosted large language model (Ollama, `qwen2.5:3b`) to generate NPC thought and belief on every turn. Generation times of up to 30 seconds per turn, measured from terminal output timestamps, were prohibitive for real-time game use. This observation drove the  decision documented in the CHANGELOG entry for 9 March 2026, replacing the LLM-based cognitive layer with a deterministic template-matching system that reduced processing time to under 5 seconds using `qwen2.5:3b`.

## 4.2 System Design

The PNE implements a Belief-Desire-Intention (BDI) agent architecture applied to NPC behaviour in narrative game dialogue. The pipeline processes a single player turn through four sequential stages — cognitive interpretation, desire formation, socialisation, and outcome selection — orchestrated by the `DialogueProcessor` class (`processor.py`). The overall data flow is: player input → cognitive layer (belief) → desire layer (want) → socialisation layer (intention) → outcome layer (consequence).

### 4.2.1 Player Input

The point of entry into the BDI pipeline is the `PlayerDialogueInput` dataclass (`player_input.py`). This structure encapsulates every attribute of the player's chosen dialogue option and is passed unchanged through all four layers so each has access to an identical snapshot of the player's dialogue choice.

The most significant attribute is `language_art`, an enum drawn from `{CHALLENGE, DIPLOMATIC, EMPATHETIC, MANIPULATIVE, NEUTRAL}`. The language art determines which of the player's four skill dimensions is used for the dice check and provides the primary categorical input to the thought-matching algorithm. Each scenario choice carries a pre-authored language art label — a design decision that keeps categorisation deterministic and author-controlled rather than inferred from natural language.

The four language arts map to four player skill dimensions — authority, diplomacy, empathy, manipulation — encoded in the `PlayerSkillSet` dataclass on a 0–10 integer scale. Skills govern the rhetorical register through which the player communicates, grounded in Cialdini's (1984) taxonomy of social influence strategies, as discussed in Chapter 3. Each `PlayerDialogueInput` also carries four continuous tone signals (`authority_tone`, `diplomacy_tone`, `empathy_tone`, `manipulation_tone`, each in [0.0, 1.0]). These are distinct from the language art: where the language art identifies which skill is checked, tone signals convey rhetorical intensity within the choice. An optional `ideology_alignment` field identifies whether a choice appeals to a specific ideological position, matched against the NPC's `social.ideology` dict in the desire layer to enable ideologically resonant choices to produce qualitatively different desire states.

The skill check is implemented in `SkillCheckSystem` (`skill_check.py`). The primary mechanism is a two-dice system: both player and NPC roll a biased d6, and `player_die >= npc_die` constitutes a success. The player die is biased by normalising the relevant skill value to [0.0, 1.0] and using this as the exponential weight parameter; the NPC dice is biased by a resistance threshold derived from personality attributes — for authority challenges, `0.3 + (social.assertion × 0.4)`, meaning assertive NPCs resist commanding approaches more strongly. The `success_probability()` method computes the pre-roll success percentage analytically by summing joint probabilities across all (player_die, npc_die) pairs, so the UI can display an estimated chance of success before the player commits. A configurable difficulty modifier (`SIMPLE: +0.15`, `STANDARD: 0.0`, `STRICT: -0.15`) is applied as an additive bias, allowing difficulty tuning without modifying NPC profiles.

### 4.2.2 Cognitive Layer

The cognitive layer receives the `PlayerDialogueInput` and the NPC model, and produces a `ThoughtReaction`: the NPC's internal, unspoken response. The `ThoughtReaction` consists of `internal_thought` (first-person reaction text), `subjective_belief` (the NPC's interpretation of the player's intent), `cognitive_state` (a snapshot of the NPC's three core cognitive attributes), and `emotional_valence` (a scalar in [-1.0, 1.0]).

The theoretical basis is Ellis's (1962) A-B-C model: the player's choice is the activating event (A); the `subjective_belief` is the mediating belief (B); and `emotional_valence` is the emotional consequence (C). The cognitive bias categories additionally draw on Beck's (1979) taxonomy of cognitive distortions — bias types such as hostile attribution, catastrophising, and black-and-white thinking are directly instantiated as template categories in the system.

The original implementation used a live LLM call to generate `internal_thought` and `subjective_belief` free-form. Log inspection revealed two problems: two LLM calls per player choice increased latency in gameplay and free-form outputs were qualitatively inconsistent — the same input and NPC configuration could produce structurally different thought formats across runs, and the `subjective_belief` text was often too abstract to serve as reliable keyword input for the downstream desire layer. These issues were identified through examination of logs such as `morrisson_moses_1732140226.json`, where belief-driven desire outputs showed poor correspondence between stated belief and the desire category selected.

The replacement architecture (`CognitiveThoughtMatcher`, `cognitive_thought_matcher.py`) performs deterministic template matching against a library of 810 pre-authored thought-pattern templates in `cognitive_thoughts.json`. Each template defines a `bias_type`, `thought_variants` and `belief_variants` (randomly selected within the winning template for variation), and a `match_weights` dictionary. The matching algorithm scores every template by evaluating `match_weights` against the current player input and NPC state: language art is a discrete lookup; numeric parameters (`npc_self_esteem`, `npc_locus_of_control`, `npc_cog_flexibility`, and the four tone signals) are gate conditions scoring their defined weight if the extracted value falls within the template's specified range. The raw score is normalised against the template's total possible weight to produce a fit score in [0.0, 1.0]. The highest-scoring template above threshold 0.35 is selected; if none clears the threshold, the system falls back to a `cynical_realism` default.

`emotional_valence` is maintained as a separate rules-based calculation: low self-esteem NPCs respond negatively to authority and manipulation; external locus associates authority cues with perceived threat; high cognitive flexibility responds positively to diplomacy and empathy. These relationships implement personality-behaviour mappings from Ellis (1962), Heider (1958), and Guimond (2006), as reviewed in Chapter 3.

### 4.2.3 Desire Layer

The desire layer corresponds to the Desire component in the BDI architecture formalised by Rao and Georgeff (1995). It receives the `ThoughtReaction` and produces a `DesireState`: `immediate_desire` (natural-language goal description), `desire_type` (categorical motivation), and `intensity` (0.0–1.0).

The four desire types — `information-seeking`, `affiliation`, `protection`, and `dominance` — were derived from the social psychology literature. The information-seeking and affiliation categories correspond to Cialdini's (1984) influence framework; protection corresponds to threat-appraisal responses in Guimond (2006); dominance maps to Goffman's (1959) face management under competitive social conditions. Restricting to four categories reflects a design trade-off: richer motivational taxonomies exist but a larger category set reduces reliability of downstream intention matching.

The desire formation algorithm implements six belief-keyword patterns in priority order: (1) **Uncertainty** — keywords such as `unclear`, `unsure`, `testing` trigger `information-seeking` (or `protection` for low self-esteem NPCs); (2) **Sincerity** — keywords such as `genuine`, `sincere` trigger `affiliation` (or guarded `information-seeking` for low-empathy NPCs); (3) **Threat** — keywords such as `manipulative`, `deceive` trigger `protection` or `dominance` depending on assertion level; (4) **Opportunism** — keywords such as `exploit` trigger elevated-intensity `information-seeking`; (5) **Ideology alignment** — player `ideology_alignment` matched against the NPC's ideology dict; (6) **Emotional valence fallback** — negative valence below -0.3 produces `protection`, positive above 0.3 produces `affiliation`.

The keyword-driven design for `subjective_belief` is the structural reason the cognitive layer must produce consistent, keyword-rich belief text. The original LLM-based cognitive layer failed this requirement because its outputs were too variable, causing the desire step to fall through all six patterns to the valence fallback disproportionately often — directly observed in output logs and what prompted the switch to the template-based approach.

After the six patterns resolve a base desire, a `BIAS_TO_DESIRE_MODIFIER` table applies a post-pattern correction: each cognitive `bias_type` from the `CognitiveThoughtMatcher` maps to a potential `desire_type` override and an additive intensity boost. Hostile attribution bias overrides desire type to `protection` and adds 0.2 intensity, modelling the empirical finding that individuals with hostile attribution tendencies interpret ambiguous social cues as threatening regardless of surface content (Graham, Hudley and Williams, 1992). This ensures two NPCs with identical personality parameters but different cognitive bias profiles produce different desire states from the same input.

### 4.2.4 Intention Layer

The intention layer corresponds to the Intention component in the BDI formalism. It receives the `DesireState` and produces a `BehaviouralIntention`: an `intention_type` from a closed canonical vocabulary, a `confrontation_level` in [0.0, 1.0], an `emotional_expression` direction, and a `wildcard_triggered` boolean.

The intention layer is implemented by `SocialisationFilter` (`social.py`), which selects from the `INTENTION_REGISTRY` — a statically defined list of `IntentionTemplate` instances (`intention_registry.py`). Every template specifies a `desire_type` it serves, `desire_keywords` that activate it, a confrontation range, emotional expression label, and optional hard gates: `wildcard_required` restricts a template to NPCs with a specific wildcard trait; `npc_conditions` specify attribute thresholds. The registry is the closed vocabulary of all valid NPC behavioural intentions — an author-enumerable design document as much as a code artefact.

The selection algorithm filters the registry to matching `desire_type`, then scores each candidate on keyword overlap (up to 0.5), confrontation band fit (up to 0.4, with partial credit for near misses), and an intensity bonus (0.1 for high-intensity desires matched to high-confrontation templates). Hard gates are evaluated before scoring; excluded templates receive -1.0.

The introduction of the intention registry (CHANGELOG, 2 March 2026) was a significant architectural revision. The original design allowed free-form intention strings, which were passed to the LLM prompt and FSM transition conditions. This created a reliability failure: the FSM could not dependably match the LLM-generated text against scenario keywords because phrasing was inconsistent across runs. Replacing free-form generation with a closed canonical vocabulary made FSM routing deterministic.

A wildcard override mechanism bypasses the normal desire-to-intention flow for extreme personality configurations. The `Inferiority` wildcard hard-triggers `Submit` whenever player authority tone exceeds 0.5; the `Napoleon` wildcard enables `Assert Dominance Aggressively`, otherwise unreachable because its confrontation range (0.8–1.0) exceeds what the normal scoring algorithm produces for typical NPC profiles. The confrontation level of the selected intention is computed by clamping the NPC's natural confrontation tendency (`assertion × 0.7 + conf_indep × 0.3`) to the template's valid range, then nudging upward by a proportion of desire intensity.

### 4.2.5 Outcome Layer

The outcome layer maps the `BehaviouralIntention` to two distinct types of outcome: an immediate `InteractionOutcome` that modifies NPC state and advances the conversation, and a `TerminalOutcome` that ends the conversation with a defined narrative consequence.

The distinction between interaction and terminal outcomes was not present in the initial design. The original implementation (CHANGELOG, 22 November 2025) defined only a single outcome type checked at every turn. Examination of output logs revealed that this produced conversations without direction or form: terminal conditions could be met at any turn, causing abrupt endings before the player had made enough choices for the outcome to feel earned. The design response was the two-tier system.

An `InteractionOutcome` carries: `stance_delta` (NPC attribute changes applied immediately); `relation_delta` (change to `world.player_relation`); `intention_shift` (optional new long-term NPC goal); and `min_response`/`max_response` scripted variants that anchor LLM generation at the extremes of the emotional valence scale. These constrain the LLM's output rather than replacing it.

A `TerminalOutcome` carries a `terminal_id` enum, a callable `condition` function evaluated against NPC state and conversation model, a `result` string, and a `final_dialogue` line. Terminal conditions are evaluated at the end of every turn; the first that evaluates to true ends the conversation. Multiple terminal outcomes per scenario node enable branching narrative endpoints that respond to accumulated conversation state.

The `judgement` tracking system (CHANGELOG, 3 March 2026) was the direct precursor to this architecture. Prior to its introduction, terminal routing was driven by raw `player_relation`, which proved too noisy — individual relation deltas could trigger a terminal outcome before accumulated choices warranted it. The judgement score (initialised at 50, shifted by dice outcomes scaled by a risk multiplier when the pre-roll odds were against the player) integrates evidence across turns more robustly, requiring sustained successful interaction to achieve positive terminal outcomes. This reflects Murray's (1997) principle that player agency requires the sense that choices have cumulative weight.

## 4.3 Information Security and Ethical Considerations

### 4.3.1 Information Security

The PNE does not collect, transmit, or persist personally identifiable information during normal operation. Conversation logs are serialised to local JSON files and not transmitted to any remote service. The Ollama language model runs as a local process, meaning no conversation content is sent to third-party servers during LLM-assisted generation.

The REST API layer (`Models/api/`) creates a network attack surface absent in the CLI-only configuration. The FastAPI server exposes HTTP and WebSocket endpoints for session creation and dialogue processing. For this project, the API is deployed exclusively in a local development environment and is not exposed to external networks. No authentication mechanism is implemented, reflecting the development-only deployment context. A production deployment would require authentication, input validation, and rate limiting — these are acknowledged as outstanding security requirements outside the scope of the current implementation.

### 4.3.2 Ethical Approach

The use of AI assistance during data collection — specifically Claude (Anthropic) to review output logs for narrative dissonance — constitutes a methodological transparency concern. All AI-assisted analysis was conducted with the developer reviewing and critically evaluating the AI's observations rather than accepting them without scrutiny. The AI was used as a pattern-recognition aid for long JSON files, not as an independent analytical authority. All NPC characters and narrative content are original creations; no copyrighted content was used without attribution.
